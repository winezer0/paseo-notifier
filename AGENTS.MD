# 项目编码&测试规范

## 本项目重点
本项目是基于Paseo的MCP接口实现的服务状态检测工具.

支持的MCP列表需要阅读文档@paseo-mcp.md

## 一、文件规范
1. 行数约束：单文件≤400行，超行按职责拆分（如nodes/edges/imports分文件）；单函数≤100行
2. 命名规则：文件小写下划线；测试文件`xxx_test.go`；函数驼峰命名
3. 导入排序：标准库 → 第三方库 → 项目内部库
4. **使用结构化和函数式编程方法，让代码保持结构优雅和可维护**

## 二、代码编写规范
1. 嵌套限制：if/for/switch嵌套不超3层，复杂逻辑拆分子函数，杜绝深层嵌套
2. 函数约束：单一职责；参数≤5个，多参数用结构体封装；禁止滥用全局变量，数据靠参数传递
3. 封装设计：用结构体聚合数据、接口定义行为，杜绝包循环依赖
4. 输出约束：print等日志输出仅用英文文本，禁止中文输出字符串
5. 常量统一：节点/边类型统一使用basetypes预定义常量，禁止硬编码字符串字面量
6. 边结构强制：所有关联边统一使用ToNode结构体存储

## 三、注释规范
1. 所有导出公开函数必须注释，格式`// 函数名 功能描述`，附带入参、返回值说明
2. 关键逻辑步骤添加精简中文注释，不写冗余无意义注释

## 四、错误处理
所有返回错误必须通过`if err != nil`捕获处理，禁止忽略error

## 五、单元测试强制规范
1. 代码配套：每个提取器模块必须配套`_test.go`单元测试文件
2. 数据规则：测试数据全部硬编码常量，不读取外部文件、不动态生成数据；断言优先完整字符串比对，不只用数量校验
3. 覆盖率：单元测试覆盖率≥90%；覆盖正常场景、边界值、异常分支、全部公开函数
4. 用例组织：采用表驱动测试 + `t.Run()`分子测试，用例名称清晰描述测试场景

## 核心强制红线
1. 全部模块必须写单元测试，测试数据硬编码无外部文件依赖
2. 不忽略错误、导出函数必带中文注释、输出仅英文文本
